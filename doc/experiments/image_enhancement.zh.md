# 图片增强实验

本文档记录桌面端图片增强和超分功能的预研判断。它是实验设计和任务跟踪文档，不代表功能已经确定会进入正式版本。

## 背景

部分低分辨率漫画页在桌面端大屏阅读时会显得模糊。AI 超分或图片增强可能改善观感，但也会带来性能、包体、跨平台打包和维护成本。

对 Mihon 代码的初步检查显示，它有常规图片处理能力，例如双页拆分、双页重排、长图拆分、旋转和解码裁切，但没有发现内置 AI 超分链路。

## 目标

- 在桌面端为低分辨率图片提供可选增强能力。
- 尽量不影响默认阅读性能。
- 先验证真实观感收益，再决定是否进入正式功能规划。

## 非目标

- 不在移动端默认启用。
- 不做实时逐帧超分。
- 不把超分作为阅读器主流程的前置依赖。
- 不在 README 中提前宣传未稳定的实验功能。

## 初步方案

- 平台范围：优先 Windows 桌面端，后续再评估 macOS 和 Linux。
- 触发方式：优先支持“增强当前图片”或“增强当前章节”，避免默认全局自动处理。
- 处理方式：异步队列处理，限制并发为 1，支持取消。
- 生效范围：只处理低于阈值的图片，例如图片宽度小于当前窗口物理像素宽度时才提示。
- 缓存策略：增强结果写入独立缓存，缓存 key 包含原图标识、模型版本、倍率和参数。
- 安全阈值：限制最大输入像素、最大输出像素和单次任务耗时。

## 技术路线

### 外部增强器优先

初版优先考虑调用外部工具，例如 Real-ESRGAN 或 waifu2x 的桌面端命令行版本。

优点：

- 不需要立刻把 ONNX、NCNN 或 GPU 推理框架深度集成进 Flutter。
- 原型成本低，可以快速验证观感收益。
- 失败时可以退回原图，不影响阅读主流程。

缺点：

- 需要处理可执行文件路径、版本检测和错误提示。
- 跨平台打包与分发策略需要单独设计。
- 用户机器上的 GPU、驱动和权限状态不可控。

### 深度集成暂缓

ONNX Runtime、NCNN、TFLite 或 OpenCV 类方案可以提供更完整的一体化体验，但会显著增加跨平台构建、二进制体积和维护复杂度。除非外部增强器原型证明收益明显，否则暂不作为第一阶段目标。

## 风险

- 桌面端也可能出现高 CPU/GPU 占用和风扇噪音。
- 部分漫画线条或文字可能被错误增强，观感反而下降。
- 章节级批量处理可能占用大量磁盘缓存。
- 内置模型或二进制工具会明显增加包体。
- 不同平台的 GPU 加速能力差异很大，问题定位成本高。

## 里程碑

1. 调研外部增强器的命令行参数、授权协议、输出格式和平台支持。
2. 实现单图增强原型，只在本地临时目录输出结果。
3. 增加独立缓存目录和缓存 key 设计。
4. 在桌面端增加隐藏或实验性设置入口。
5. 增加“增强当前图片”的手动触发入口。
6. 验证低分辨率图片、正常分辨率图片、文字页和横向双页的实际效果。
7. 根据测试结果决定是否支持“增强当前章节”。

## 暂缓条件

- 单图处理速度明显影响阅读节奏。
- 增强结果对文字清晰度或线条稳定性提升不明显。
- 包体、依赖、授权或跨平台构建成本过高。
- 用户需要额外安装复杂依赖才能使用。

## 当前结论

图片增强可以作为桌面端实验功能继续预研，但不适合移动端默认启用，也不适合在功能稳定前写入 README 的正式功能介绍。
